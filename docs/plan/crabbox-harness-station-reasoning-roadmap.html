Roadmap thesis

Crabbox is the proof substrate first, and a reasoning host only by explicit choice.

Sell trust in remote work, not another agent brain. Make every Codex, OpenClaw, maintainer, and CI-like run more grounded, replayable, and billable. A box that reasons is just an explicit station workload with evidence, budgets, policy, and teardown.

thin control plane harness-aware jobs station leases opt-in reasoning access proof and compliance

The decision

Keep the default contract small: lease a box, sync the real checkout, run a command, stream output, capture visual and file evidence, release. That is the Peter-aligned core.

Reasoning is valid, but it belongs in an explicit station workload chosen by the repo or user. Crabbox owns lifecycle, policy, spend, sync, logs, artifacts, and compliance evidence — never the model loop or agent brain.

The trap: if "reasoning box" becomes the default story, Crabbox competes with the agents it should amplify — heavier, harder to trust and secure, less useful to today's users.

Principles (Peter test)

Crabbox is not the brain

Codex, OpenClaw, Claude, and repo workflows reason and plan. Crabbox provides grounded execution and evidence.

The repo owns runtime weight

Dependencies, services, devcontainers, CI hydration, and agent loops stay repo-owned. Default boxes stay light.

Dirty sync is the moat

Run the exact local checkout, including uncommitted changes, on real remote capacity with reproducible evidence.

Proof beats autonomy theater

Make every run reviewable: command, logs, timing, screenshots, JUnit, artifacts, harness, grounding, compliance.

Peter test: if a feature does not make the existing remote loop faster, more trustworthy, easier to review, cheaper, or more billable, it does not belong in the next milestone.

Three modes, one substrate

Each mode reuses the same run pipeline. The line not to cross is in the last column.

Mode Use when Crabbox provides Must not provide by default
Default runner One command, short feedback, CI-like validation, screenshots, artifacts. Lease, sync, run, logs, timing, artifacts, release. Agent brain, model credentials, scenario state.
Harness job (Phase 1) A run needs intent, scope, plan evidence, required artifacts, compliance status. Validation, light grounding, compliance report, proof artifacts. Interpreting the plan as autonomous agent policy.
Station (Phase 2) A repo-owned process needs a warm lease for hours with logs and evidence. Lifecycle, heartbeats, WebVNC, logs, artifacts, TTL, idle timeout. Bundled LLM runtime or hidden orchestration.
Reasoning station (Phase 3) A chosen agent runs close to the workspace with bounded access. Reasoning-access policy, audit trail, spend controls, compliance evidence. Unbounded model access, coordinator credentials, default bootstrap weight.

Phase 1: Harness-aware jobs

Highest value first: wrap existing runs with intent, scope, grounding, and a compliance verdict — no change to what a box is. A harness is Markdown with YAML frontmatter (version, template, job, plan_file, scope, validate, compliance). 

crabbox harness validate HARNESS.md
crabbox job run --harness HARNESS.md --index light regression
crabbox run --harness HARNESS.md -- npm test

Ground

Hashes, git SHA/branch, dirty summary, sync scope, selected paths, command and job metadata under .crabbox/grounding/.

Evidence

Copy harness.md, grounding.json, compliance-report.md, and .json into local run artifacts.

Verdict

Exit 0 is necessary but not sufficient. Missing required evidence turns the result nonzero after the report is written.

MVP cut line: no vector index, no LLM judging, no scenario database, no autonomous state machine, no agent memory, no new remote daemon. Boring on purpose: parse, validate, ground, run, collect, report.

Phases 2 & 3: Station and reasoning station

Build these only after harness runs prove useful, reusing the same harness and compliance model. A station is an explicit long-lived lease around a repo-owned command; a reasoning station adds scoped model access under policy. Neither adds an LLM runtime to Crabbox core. 

crabbox station start --job agent-loop --harness HARNESS.md \
  --command "scripts/agent-loop.sh" --ttl 10h --idle-timeout 45m
crabbox station status|logs|stop <id-or-slug>

Crabbox owns

  • Lease lifecycle, warm pools, teardown.
  • Dirty sync and workspace prep.
  • Credential delivery policy and redaction.
  • TTL, idle timeout, spend caps, heartbeats.
  • Logs, artifacts, timing, compliance reports.

The workload owns

  • Model choice and prompt loop.
  • Planning, tool use, edits, review strategy.
  • Repo-specific setup and long-running command.
  • Failure recovery inside the agent loop.
  • How to interpret test failures.
Non-negotiable: coordinator credentials never enter the box. Model credentials, if used, are scoped workload credentials with explicit TTL, redaction, and audit metadata — never written to logs or reports.

How to build it

A thin layer over the existing run pipeline: resolve intent before the run, upload small grounding files once the workspace exists, write compliance evidence after results and artifacts are collected.

New / extended files

  • harness.go: parse frontmatter, validate, hash; grounding + compliance builders and renderers.
  • config.go: HarnessConfig and JobConfig.Harness.
  • job.go / run.go: --harness, --index none|light, grounding upload, compliance artifacts.
  • timing.go: harness metadata and artifact kinds.
  • index.js + openclaw.plugin.json: two bounded plugin tools.

Execution flow

  1. Parse Markdown, split frontmatter, reject unknown critical fields.
  2. Merge job defaults with file frontmatter; CLI flags win.
  3. Ground (default light when a harness is present).
  4. Upload to remote .crabbox/grounding/ after sync.
  5. Run the command unchanged.
  6. Evaluate exit, plan, artifacts, JUnit, evidence.
  7. Report locally and into timing metadata.
type HarnessMetadata struct {
    Path        string `json:"path"`
    Template    string `json:"template,omitempty"`
    HarnessHash string `json:"harnessHash"`
    PlanHash    string `json:"planHash,omitempty"`
    Index       string `json:"index"`
    Status      string `json:"status"`
}

Parser and validator

Data types, frontmatter parser, validation, hashes, crabbox harness validate, tests. No run change.

Grounding artifacts

Local grounding, upload to remote, copy into .crabbox/runs/<id>/, timing JSON metadata.

Compliance enforcement

Markdown/JSON reports for success and failure; missing required evidence becomes a nonzero result.

OpenClaw tools and docs

crabbox_harness_validate, crabbox_job_run_with_harness, plugin tests, harness docs.

Milestone gates and revenue

MilestoneShip only ifStop ifSells
Harness jobs Users get better PR evidence with no workflow rewrite. It needs a daemon, model call, DB, or heavier box. Trust: searchable history, compliance, branded proof.
Station Harness runs show demand for long-lived workspaces. It becomes an agent framework, not lease supervision. Control: status, logs, WebVNC, stop, spend caps.
Reasoning station Power users run agent loops and will pay for safe lifecycle. Crabbox must choose the model or interpret success. Safe autonomy hosting: warm pools, policy, audit.

Anti-goals and test plan

Anti-goals

  • No hidden agents: no implicit LLM runtime or default model credentials in ordinary boxes.
  • No thicker default box for users who only need sync, run, proof, release.
  • OpenClaw stays an integration, not the product identity.
  • Station means a warm lease with controls, not autonomy.

Test plan (Phase 1)

  • Parse/validation/merge unit tests.
  • job run --dry-run --harness expansion and run --harness option tests.
  • Fake-SSH runs: success, command failure, missing artifact/plan, JUnit failure.
  • Timing JSON includes harness metadata and artifacts.
  • Plugin tests for schemas, argv, timeout, disabled-tool guard.

Bottom line: ship harness jobs to monetize proof in the existing flow; add station for reliable long runs; add reasoning station last as an explicit, policy-bound profile — without turning Crabbox into the brain.
