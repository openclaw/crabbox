Roadmap thesis

Crabbox should be the proof substrate first, and a reasoning host only by explicit choice.

The strongest path is to sell trust in remote work, not to cosplay as another agent brain. Crabbox should make every Codex, OpenClaw, maintainer, and CI-like run more grounded, observable, replayable, and billable. A box that can reason is useful only when it is an explicit station workload with evidence, budgets, policy, and teardown.

thin control plane harness-aware jobs station leases opt-in reasoning access proof and compliance

The decision

Keep the default Crabbox contract small: lease a box, sync the real checkout, run a command, stream output, capture visual and file evidence, and release. That is the Peter-aligned product core.

The new idea is valid: some boxes should be able to reason. The important distinction is where that capability lives. Reasoning belongs in an explicit station workload selected by the repo or user. Crabbox core should own lifecycle, policy, spend, sync, logs, screenshots, artifacts, and compliance evidence. It should not own the model loop, prompt loop, or agent brain.

The trap: if "reasoning box" becomes the default story, Crabbox starts competing with the agents it should amplify. That makes the product heavier, harder to trust, harder to secure, and less obviously useful to today's users.

What is good, what is bad

Good

Harness as proof wrapper

This fits Crabbox. It wraps existing runs with intent, scope, grounding, and compliance without changing what a box is. It makes current workflows more valuable immediately.

Good

Station as warm lease

This also fits. It is a lifecycle product: keep the workspace warm, supervise a repo command, collect evidence, and stop cleanly.

Bad

LLM inside core

This is the product fork to avoid. If Crabbox owns prompts, agent state, model routing, and reasoning policy by default, it stops being the neutral substrate.

Bad

Heavy default boxes

Baking model clients, project runtimes, vector stores, or agent supervisors into ordinary boxes slows down the main value loop and creates support burden before revenue proves it is worth it.

Peter's school of thought

1. Crabbox is not the brain

Codex, OpenClaw, Claude, custom scripts, and repo workflows do reasoning and planning. Crabbox provides grounded execution and evidence.

2. The repo owns runtime weight

Project dependencies, services, devcontainers, Nix, mise/asdf, CI hydration, and agent loops remain repo-owned choices. Default boxes stay light.

3. Dirty sync is the moat

Crabbox should excel at running the exact local checkout, including uncommitted changes, on real remote capacity with fast feedback and reproducible evidence.

4. Proof beats autonomy theater

The product should make every run more reviewable: command, logs, timing, screenshots, JUnit, artifacts, harness, grounding, and compliance reports.

Peter test: if a feature does not make today's remote execution loop faster, more trustworthy, easier to review, cheaper to operate, or more billable, it should not be in the next milestone.

Two product lanes

Lane A: Proof substrate

This ships first. Harness-aware jobs add intent, plan, scope, validation, grounding, and compliance evidence around the existing job run and run flow.

  • Generic harness language, not ai_harness.
  • Light grounding by default when a harness is present.
  • Compliance reports become first-class run artifacts.
  • OpenClaw gets two bounded tools, not raw shell passthrough.

Lane B: Reasoning station

This comes later and remains opt-in. It is a kept-warm station lease with explicit reasoning access and a repo-owned long-running command.

  • Crabbox can host reasoning workloads without becoming the reasoning product.
  • The user or repo chooses the model client and loop command.
  • Crabbox enforces policy, budgets, lifecycle, evidence, and shutdown.
  • No coordinator credentials or implicit model credentials live in default boxes.

Phase 1: Harness-aware jobs

This is the highest-value move because it improves the current flow immediately. It gives maintainers and agents a structured way to say what is being tested, why it matters, what files are in scope, and what evidence must exist before a run counts as successful.

Revenue value: better run yield, stronger PR evidence, fewer wasted long runs, and a clear paid tier around compliance history, branded reports, and searchable proof. 
crabbox harness validate HARNESS.md
crabbox job run regression --harness HARNESS.md --index light
crabbox run --harness HARNESS.md -- npm test

Harness file

Markdown with YAML frontmatter: version, template, job, plan_file, scope, validate, and compliance.

Grounding

Write hashes, git state, branch, dirty summary, sync scope, selected paths, command metadata, and job metadata under .crabbox/grounding/.

Evidence

Copy harness.md, grounding.json, compliance-report.md, and compliance-report.json into local run artifacts.

MVP cut line: no vector index, no LLM judging, no scenario database, no autonomous state machine, no agent memory, and no new remote daemon. The first release should be boring: parse, validate, ground, run, collect, report.

Phase 2: Station mode

Station mode should not be a hidden change to run. It should be an explicit long-lived lease with a harness workspace, repo-owned command, heartbeats, logs, WebVNC, artifacts, and compliance reports. 

crabbox station start \
  --job agent-loop \
  --harness HARNESS.md \
  --command "scripts/agent-loop.sh" \
  --ttl 10h \
  --idle-timeout 45m

crabbox station status <id-or-slug>
crabbox station logs <id-or-slug>
crabbox station stop <id-or-slug>
Station guardrail: station is lifecycle infrastructure. It does not add an LLM runtime to Crabbox core. The long-running process is repo-owned and auditable.

Phase 3: Reasoning station

A reasoning station is the safe version of the "box that can think" idea. It is a station profile where the box may receive model credentials or access to a model gateway under explicit policy. The reasoner is a workload, not a Crabbox subsystem.

What Crabbox owns

  • Lease lifecycle, warm pools, and teardown.
  • Dirty sync and workspace preparation.
  • Credential delivery policy and redaction boundaries.
  • Spend caps, TTL, idle timeout, and heartbeats.
  • Logs, screenshots, artifacts, timing, and compliance reports.

What the workload owns

  • Model choice and prompt loop.
  • Planning, tool use, code edits, and review strategy.
  • Repo-specific setup and long-running command.
  • Failure recovery inside the agent loop.
  • How to interpret test failures and propose fixes.
Recommended naming: use reasoning-station, agent-station, or profile: reasoning. Avoid making "reasoning box" the default product name because it weakens the thin-substrate story.

The reasoning station contract

The best version of the reasoning box is not a magic box. It is a normal station with one extra capability: explicit access to a reasoning service under policy. That capability should be visible in config, visible in run evidence, limited by budget, and removable without changing ordinary Crabbox behavior.

Inputs

  • --harness with scope and compliance expectations.
  • A repo-owned loop command such as scripts/agent-loop.sh.
  • An explicit model access profile or gateway token.

Controls

  • TTL, idle timeout, spend cap, and max log volume.
  • Egress allowlist and secret redaction boundaries.
  • Manual stop, heartbeat failure stop, and compliance failure status.

Outputs

  • Plan versions, command log, screenshots, and artifacts.
  • Model access receipts without leaking secrets.
  • Compliance report tying the loop back to harness scope.
Non-negotiable: coordinator credentials never enter the box. Model credentials, if used, are scoped workload credentials with explicit TTL, redaction, and audit metadata.

Decision matrix

Mode Use when Crabbox provides Must not provide by default
Default runner One command, short to medium feedback, CI-like validation, screenshots, artifacts. Lease, sync, run, logs, timing, artifacts, release. Agent brain, model credentials, long-lived scenario state.
Harness job A run needs intent, scope, plan evidence, required artifacts, and compliance status. Harness validation, light grounding, compliance report, proof artifacts. Interpretation of the plan as autonomous agent policy.
Station A repo-owned process needs a warm lease for hours with logs and evidence. Lifecycle, heartbeats, WebVNC, logs, artifacts, TTL, idle timeout. Bundled LLM runtime or hidden orchestration.
Reasoning station A chosen agent/runtime should run close to the workspace with bounded access. Explicit reasoning access policy, audit trail, spend controls, compliance evidence. Unbounded model access, coordinator credentials, or default bootstrap weight.

Implementation roadmap

Harness schema and validation

Add frontmatter parsing, validation, and config merge from job defaults plus file frontmatter. Keep the first schema small and generic.

Run integration and light grounding

Add --harness and --index none|light to run and job run. Upload lightweight grounding metadata to the remote workdir and copy it back into local run artifacts.

Compliance reporting

Produce Markdown and JSON compliance reports. Required harness evidence, required artifacts, JUnit, and plan evidence decide whether a successful command remains successful.

OpenClaw plugin tools

Add crabbox_harness_validate and crabbox_job_run_with_harness with typed schemas, disabled-tool guards, timeout handling, and capped output.

Station preview

Add the minimal station lifecycle once harness jobs prove useful: start, status, logs, stop, TTL, idle timeout, and repo-owned command execution.

Reasoning station preview

Add an explicit station profile that can receive model access under policy. Treat this as a premium capability and keep all model/runtime choices outside Crabbox core.

How to build it

Build the feature as a thin layer over the existing run pipeline. The harness should not create a second executor. It should resolve intent before the run, upload small grounding files after the workspace exists, and write compliance evidence after command/results/artifact collection.

New core files

  • internal/cli/harness.go: parse Markdown frontmatter, validate schema, compute hashes.
  • internal/cli/harness_grounding.go: build light grounding JSON from repo, git state, sync scope, command, and job metadata.
  • internal/cli/harness_compliance.go: evaluate required evidence and render Markdown/JSON reports.
  • internal/cli/harness_test.go: parser, validation, merge, and report tests.

Existing files to extend

  • internal/cli/config.go: add HarnessConfig and JobConfig.Harness.
  • internal/cli/job.go: add --harness, --index, dry-run expansion, and job-default merge.
  • internal/cli/run.go: add --harness, resolve harness, upload grounding, collect compliance artifacts.
  • internal/cli/timing.go: add harness metadata and artifact kinds.
  • index.js and openclaw.plugin.json: add two bounded plugin tools.

Harness execution flow

Stage Where Behavior
1. Parse harness validate, run, job run Read Markdown, split YAML frontmatter from plan body, validate known fields, reject unknown critical fields.
2. Merge job.go Merge job-level harness: defaults with file frontmatter. CLI flags win over job defaults.
3. Ground run.go Default --index light when a harness is present. Record git SHA, branch, dirty summary, selected paths, hashes, command, job, run label, and sync settings.
4. Upload run.go using the existing SSH input/upload pattern After workdir is known and sync completes, upload harness.md and grounding.json under remote .crabbox/grounding/.
5. Run existing command path Do not change command execution semantics. Harness wraps the run; it does not interpret or mutate the command.
6. Evaluate after JUnit/results/artifact collection Check command exit, required plan file, required artifacts, JUnit status, and evidence presence. Failed commands still write reports when context exists.
7. Report run_artifacts.go / local artifact path helpers Write local harness artifacts and append harness, grounding, compliance-report, and compliance-json entries to timing metadata.
Exit rule: with a harness, command exit 0 is necessary but not sufficient. Missing required evidence should convert the CLI result to nonzero after writing the compliance report.

Data model

type HarnessConfig struct {
    Path       string
    Template   string
    PlanFile   string
    Scope      []string
    Validate   HarnessValidateConfig
    Compliance HarnessComplianceConfig
}

type HarnessMetadata struct {
    Path        string `json:"path"`
    Template    string `json:"template,omitempty"`
    HarnessHash string `json:"harnessHash"`
    PlanHash    string `json:"planHash,omitempty"`
    Index       string `json:"index"`
    Status      string `json:"status"`
}

type TimingReport struct {
    // existing fields...
    Harness *HarnessMetadata `json:"harness,omitempty"`
}

Keep the public schema small. More fields can be added to JSON later, but removing or renaming proof fields will hurt agent workflows and dashboards.

Station build path

Station should be implemented only after harness runs are useful, and it should reuse the same harness and compliance model. Technically, the first station can be a supervised composition of existing primitives: warmup --keep, optional hydration, remote command start, heartbeats/status, logs, and stop.

First implementation

  • Create a lease with existing warmup/acquire code.
  • Resolve harness and upload grounding exactly like run.
  • Start a repo-owned command with a generated station ID and log path.
  • Expose status/logs/stop by reading lease/run state and remote log files.

Defer deliberately

  • No new daemon in the first station release.
  • No semantic task scheduler.
  • No model-specific orchestration.
  • No hidden restart policy beyond explicit user settings.

Reasoning station build path

Implement reasoning station as policy and credential plumbing on top of station, not as a new executor. The initial version can be a named profile that allows a scoped model gateway token to be exposed to the repo-owned command.

Config

station.reasoning.enabled, model gateway name, allowed env vars, egress profile, max TTL, max spend, and required harness compliance.

Runtime

Inject only scoped workload credentials. Record that access was granted, but never write token values to logs, timing JSON, reports, or artifacts.

Evidence

Compliance reports should say which reasoning profile was used, which command ran, which budget applied, and whether required proof was produced.

Milestone gates

Milestone Ship only if Stop if
Harness-aware jobs Existing users can attach intent and get better PR evidence with no workflow rewrite. The design requires a daemon, model call, remote database, or new box bootstrap weight.
Station mode Harness runs show demand for long-lived workspaces and users ask for logs/status/stop around them. Station becomes an agent framework instead of lease supervision and evidence capture.
Reasoning station Power users already run agent loops manually and will pay for safer lifecycle, policy, and proof. The feature requires Crabbox to choose the model, own prompts, or interpret task success semantically.

Revenue sequence

Now: proof and compliance

Sell better evidence for PRs, autoreview, maintainers, and teams. The package is trust: searchable run history, compliance reports, branded proof, and fewer ambiguous failures.

Next: reliable long runs

Station mode increases lease duration and user trust for multi-hour workflows. The package is control: status, logs, WebVNC, stop, replayable artifacts, and spend boundaries.

Later: reasoning access

Reasoning stations can be priced around long-lived compute, warm pools, model-access policy, audit logs, and high-value agent workloads. The package is safe autonomy hosting, not another chatbot.

Anti-goals

Do not ship hidden agents

No implicit LLM runtime, hidden coordinator, or default model credentials inside ordinary Crabbox boxes.

Do not thicken the default box

Avoid increasing bootstrap weight for users who only need sync, run, proof, and release.

Do not make OpenClaw the product identity

OpenClaw should remain an important integration. Crabbox should be generic infrastructure for maintainers and agents.

Do not confuse station with autonomy

Station means a kept-warm lease with lifecycle controls. Autonomy is supplied by the repo-owned command.

Test plan for the first implementation phase

  • Go tests for harness frontmatter parsing, validation errors, and config merge from job defaults plus file frontmatter.
  • Go tests for job run --dry-run --harness command expansion and run --harness option validation.
  • Fake-SSH run tests for success, command failure, missing required artifact, missing plan file, and JUnit failure.
  • Timing JSON tests proving harness metadata and artifact entries are included.
  • Plugin tests for tool registration, schemas, argv shape, timeout behavior, and disabled-tool guard behavior.
  • Docs checks covering harness files, compliance reports, and station mode preview.

Bottom line: ship Harness-Aware Jobs first because they monetize trust and proof in the existing flow. Add Station next because it monetizes reliable long-running work. Add Reasoning Station later as an explicit, policy-bound station profile for users who want boxes that can reason without turning Crabbox itself into the brain.
